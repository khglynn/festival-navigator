**Date:** 2026-07-12 · **Branch:** v31-polish · **Scope:** copy only (proposal, no edits)

## The 3 highest-impact changes

1. **"Likes" is the wrong noun for picks — rename it everywhere.** The tools
   menu says *Export likes* / *Bulk paste likes*, and the export body is
   literally `Person: Artist (Must)` — those are **picks**, not likes. This
   breaks the `picked / must / notes / fest` vocabulary rule *and* collides
   head-on with Spotify's own "likes" (liked songs) two rows away in the same
   Settings list. → **Export picks / Bulk paste picks.** (Details in the Tools
   and Vocabulary sections.)

2. **Create step 2 headline + sub (Kevin's flag).** "NOW YOU" over "Your name
   becomes your color." reads as a riddle — the sub asserts a metaphor instead
   of explaining the mechanic. Fix the sub to say what actually happens: you get
   a color, and the crew sees your picks in it.

3. **Wait states have no stage-lights energy (Kevin's flag).** "Creating…",
   "Joining…", "Researching…", "Building the image…" are all bare gerunds. With
   a loader component landing, give the create moment (and its siblings) copy
   with a pulse to it. Proposals in the Wait states section.

Quick win alongside these: **"Later — to the wall" → "Later"** (Kevin's flag).

**Add a festival research** — "Researching — this takes ~20 seconds…":
- **KEEP the "~20 seconds" honesty** (it's the one place a real duration helps —
  it's a genuinely long wait and the number sets expectations). Optionally warm
  the verb: **"Digging up the lineup — about 20 seconds…"**.

**Spotify scan** — "Scanned {n} of {m} liked songs…" / "Scanned library + {n}

- **KEEP.** These are live-count progress, which is the *right* pattern for a
  long scan — a determinate count beats any cute copy here. Don't loader-ify
  these.

- **KEEP.** Same reasoning — determinate progress, names the current artist.

real progress number for a spinner-with-vibe would be a downgrade.

## Vocabulary audit

The rule (CLAUDE.md): UI vocabulary is exactly **picked / must / notes / fest**.

**Violation — "likes" for picks.** Three tool strings call picks "likes":
- `EXPORT LIKES` (tools.js:44) → **EXPORT PICKS**
- `BULK PASTE LIKES` (tools.js:95) → **BULK PASTE PICKS**
- Settings rows "Export likes" / "Bulk paste likes" (settings.js:652–653) →
  **"Export picks" / "Bulk paste picks"**

Why this matters beyond the rule: the export payload is `Person: Artist (Must)`
— the noun for that is unambiguously **pick**. And "likes" is *already taken* in
this app by Spotify (liked songs), sitting two rows down in the same list. A user
who sees "Export likes" and "Refresh my likes" in one screen can't tell they're
about two different systems. Renaming picks-export to "picks" removes the
collision and honors the vocabulary rule. No code identifiers need to change —
this is display copy only (`exportLikesText`, `openExportLikes` etc. can keep
their names).

**Defensible "likes" — Spotify's "Refresh my likes" (settings.js:957).** This
one refers to Spotify liked songs, so "likes" is domain-correct. But it actually
rescans liked songs *and* followed artists, so "likes" undersells it slightly.
Optional tighten: **"Rescan my Spotify"** or **"Refresh library"** — matches the
"Your library" card header right above it. Low priority; not a rule violation.

**No stray star / favorite / bookmark / save-for-the-pick-concept found.** The
"pick / picked / must" language is otherwise consistent across cards, counts,
playlist, export bodies, and the how-it-works lessons. Good discipline.

**"How it all works" vs "How it works".** The coach-mark link says "How it all
works"; the Settings row and subview say "How it works". Same destination — pick
one. Recommend **"How it works"** everywhere.
